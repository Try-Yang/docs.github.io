    # branch管理

    branch作为我们再git操作中最常碰到的概念，有很多操作方法

    # 查看分支

    git branch
    1

    # 创建分支

    git branch dev # 创建dev分支
git checkout dev  # 切换dev分支
git checkout -b dev # 合成命令，创建同时切换一步完成
    1
    2
    3

    创建分支，默认情况下就是根据当前的 commits 链，以 HEAD 为交叉点重新延伸一条branch出来，然后将工作空间也同步的切换为创建后的分支

    当然关于 checkout这个命令还有很多的参数可供选择，这里暂时只讨论 迁出分支指定版本的做法，因为默认情况下是HEAD版本

    git checkout master # 签出 master分支的head
git checkout tagname # 在当前分支上，签出 tagname的版本
git checkout master file # 取消对文件 file的修改
git checkout commitId # 签出指定的commitId版本
    1
    2
    3
    4

    # 删除分支

    git checkout -d dev # 删除dev分支
    1

    注意这里，如果当前分支没有merge操作的话再删除操作的时候会失败

    # 合并分支

    git merge dev # 在当前分支合并dev分支
    1

    合并是从当前分支和目标分支的 交叉commit开始，把目标分支 commit的路径上的所有 commit 的内容一并和当前 分支的commit 合并生成一个新的 commit。

    # 冲突解决

    merge的过程中不可能总是一帆风顺的，如果说两个分支改了不同的文件，那么合并后自然没什么问题，如果改了同一个文件，但是修改的不是同一行，merge也会自动处理这种差异，但是如果说同时处理了同一个文件的同一行信息的话，那就不可避免的产生冲突，git不知道以哪个分支的为准了，这时候就会产生冲突。

    # 取消冲突

    如果这次提交发现有冲突后，决定放弃，那么可以使用 git merge --abort来手动取消它

    # 特殊的合并情况

    如果 再执行 merge的时候，当前 的 commit 和 目标分支 不存在分叉，也就是 当前的 commit 比目标领先，那么就不需要再创建 新的 commit了，此时 git 什么都不会做，反之，如果是落后状态，git会直接移动 HEAD以及他所指向的branch到目标commit。

    # Pull Request开发模式

    关于如何利用分支来进行高效的对多人合作开发，一般情况下是每当有一个新功能的时候每个开发人员根据自己 负责的模块建立自己的开发分支，当大家的模块都开发完毕后，再合并到一个 功能分支 去进行测试，没问题的话再合并到master分支进行版本上线。这就是一个简单的多人开发模式。

    这种模式的核心是

    • 每一个新功能都建立一个 branch 出来，然后所有的开发人员将自己的模块完成后合并到这个功能分支上
    • 功能分支测试无问题，合并到主分支上，然后可以选择删除这个功能分支

    以上分支开发模式已经很好了，但是我们一般在上线前都会进行代码评审，这时候 假设同事A拉下来最新的功能分支代码，提出了一些问题，然后通知开发进行修改，然后再推上去，同事b又提出了一个问题，我们再进行同样的流程。虽然最后可以以最优的代码合并到master，但是再代码评审环节多少有点繁琐，此时就可以使用pull request了，在远程的代码空间简历一个新的 讨论区，别人可以再对应的 pull request 中查看自己的commits，同时给出评论，然后开发者可以根据 审查者的意见把最新的commits推送上来，这时候 pull request也会对应更新，直到最有的审查都通过后，点击合并按钮，git 会自动将当前分支合并到 目标分支上去，极大的简化了多人开发中代码审查的问题
