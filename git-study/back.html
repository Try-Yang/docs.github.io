# 什么是版本回退？

就是将已经修改好的工作区内容回归到修改点之前，不管是回退一个 commit，还是一个文件，都可以被叫做回退操作

# 工作区和暂存区和版本库

# 工作区

我们的工作目录，也就是操作项目文件的地方，是我们可以直接修改的地方

# 暂存区

保存了下一次 commit的文件信息列表，它位于 根目录.git文件夹下的 index文件，一般我们都会称为暂存区

# 版本库

也就是我们上面提到的 .git隐藏文件夹，他里面存了很多东西，包括git hooks的配置，暂存区的信息等等，不要随便操作

# 如何取消上一次的提交？

经常会碰到这样一种情况，我们再提交代码的时候，commit完成后发现，有段代码有问题，这时候怎么办呢？

  • 修改错误代码，重新提交，但是会多一个提交记录

  • 使用 git commit --amend命令

    这时候，git不会再创建新的commit，而使将当前commit的内容和暂存区的内容合并起来 创建一个新的 commit，然后把当前commit替换掉，当然有时候我们仅仅是commit的信息写错了，想更改一下信息，也可以直接使用这个命令。

一般再控制台会直接显示上述内容，直接使用 vim 输入模式进行修改保存退出即可

# 修改第 n - 1 次提交呢？

有可能我们是提交了好几次后才发现前面有一次的提交有问题，那么该 怎么办呢？粗鲁一点话就直接修改后再提交，但是如果刚好有一个人根据 commit信息使用了对应的commit签出了一个新的分支，那么就会出现问题。

这时候需要使用 git rebase -i 了，使用变基操作来实现,通过这个命令可以合并提交记录，同时也可以进行提交记录的修改，他就像一个控制器。先说说怎么使用把

git log --oneline

08cc31e master second commit
80928c3 first commit
# 当前分支有以上四个提交记录，如果要修改第二个的话 acc217f，此时使用rebase进行操作
git rebase -i HEAD^^ # 执行命令，HEAD之前已经提过，代表当前的commit
# 此时命令行会进入如下页面，这时候会顺序列出 commit链，此时我们要修改的是acc217f的commit，这时候将pick 改为edit即可，注意这里是vim模式，当然还有很多其它模式，注释中已经列出来了，先不做介绍

# 直接丢弃提交

有时候发现我们的有一个commit写的问题特别多，修改的代价很大不如直接删除，这时候修改还不如完全删除。

# git reset --hard

通过这条命令，可以撤消对应的commit

git reset --hard HEAD^^^ 表示要撤消哪一个 ，对应的也可以理解为恢复为某个commit的父提交，比如说撤消当前 就是 HEAD^,恢复到他的 上一个提交去

git reset --hard HEAD^^

# 个人独立开发分支

如果当前错误内容不影响其他人，那么将对应的commit 修改删除，然后强制push即可

git push origin dev -f

# 已经进行功能分支合并了

这时候肯定不能使用强制push，否则很容易产生冲突。影响到别人的代码。只能通过添加新的提交来替代之前的提交

git revert HEAD^^ 通过这个命令来修改出错的commit，然后重新提交
